Step 3 — Imbalanced Data
Why SMOTE?

Chosen Technique: SMOTE (Synthetic Minority Oversampling Technique)

With only 140 fraud cases in the training set (0.67%), a model trained on raw data would be biased toward the majority class. Three approaches were considered: Undersampling (discard normal transactions), class_weight='balanced' (penalize misclassifying fraud more), and SMOTE (generate synthetic fraud examples).

SMOTE was chosen because it preserves all normal transaction data (unlike undersampling) and provides richer training signal than just reweighting. SMOTE generated synthetic fraud examples by interpolating between real fraud cases in feature space, expanding the fraud class from 140 samples to 20,860 — matching the normal class perfectly.

Step 6 — Conclusion
Final Analysis

Which model performed best, and why?

Random Forest outperformed Logistic Regression on every metric: Precision (1.0000), Recall (1.0000), F1-Score (1.0000), and ROC-AUC (1.0000). It achieved a perfect confusion matrix with zero False Positives and zero False Negatives on the test set — catching all 60 fraud cases while raising no false alarms.

Logistic Regression served as a strong interpretable baseline with F1 = 0.9421 and ROC-AUC = 0.9997. It missed 3 fraud cases and produced 4 false alarms — still excellent, but inferior to Random Forest.

Why F1-Score, not Accuracy? Because the dataset is imbalanced (only 0.67% fraud), a model that always predicts "Normal" would achieve over 99% accuracy while catching zero fraud. Accuracy is misleading here. Recall is critical — missing a fraud transaction has real financial and reputational cost — while Precision matters to avoid flagging legitimate customers. F1-Score balances both.

SMOTE was the key preprocessing decision that enabled strong performance. By generating synthetic fraud examples and balancing the training classes from 140 vs. 20,860 to a perfectly balanced set, both models could learn meaningful fraud patterns rather than defaulting to the majority class.
